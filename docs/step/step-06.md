# Step 06: Perft による総合検証の実装

## このステップの目的

合法手生成、通常着手、強制パスまで揃った段階で、Perft を実装して局面遷移全体を既知値で検証できるようにする。
この段階では石数集計や勝敗判定には進まず、ゲーム進行の総合的な正しさを独立して確認できる状態を作ることを目的とする。

## このステップで行うこと

- `perft(board, depth, mode)` を実装する
- `PerftError` 型を実装する
- 必要なら内部再帰 helper を実装する
- 初期局面の既知値に対する Rust テストを追加する
- 強制パスを含む局面での Perft テストを追加する
- `README.md` と `Makefile` に長時間検証用の入口を追加する

## 導入対象

- `perft(board, depth, mode)`
- `PerftError`
- Perft 用の内部再帰 helper
- 長時間検証用の `Makefile` / `README` 記述

## 実装開始時点の不足

Step 05 時点では、合法手生成、通常着手、強制パス、局面種別判定までは実装されているが、それらをまとめて既知値で検証する総合テストは未実装である。
このため、後続の石数集計や勝敗判定へ進む前に、Perft によってゲーム進行の全体整合性を確認する必要がある。

## 実装方針

- `perft(board, depth, mode)` は `Result<u64, PerftError>` を返し、`mode` のデフォルト値は 1 とする
- 公開 API の `mode` は `1` / `2` の数値で受け取り、内部では専用 enum に変換して扱う
- `mode` が `1` / `2` 以外の場合は `PerftError::InvalidMode` を返す
- 深さ 0 は葉として 1 を返す
- 合法手がある局面では、全合法手に `apply_move_unchecked` を適用して再帰する
- 合法手がない局面では `board_status` を使い、`ForcedPass` なら `apply_forced_pass` で進めて再帰する
- モード1ではパスを 1 手として扱う
- モード2ではパスを 0 手として扱う
- `Terminal` なら指定深さに達していなくても葉として 1 を返す
- 実装方針の具体化では `ref` 配下の参考実装を参照する
- 通常テストで常設確認する既知値は初期局面の深さ 1 から 8 までとする
- 深さ 9 から 15 の既知値確認は長時間検証として `Makefile` と `README` から手動実行できるようにする
- 深さ 1 から 8 までは両モードで同じ値になる前提を共有する
- 深さ 9 以降はパスの数え方の差によりモード1とモード2の値が分岐する
- 強制パス局面については固定局面を使ってモード差を確認する
- 長時間検証用コマンドでは、ルートの合法手単位で進捗表示を行う
- 進捗表示はデバッグ・手動確認用に限定し、`perft(board, depth, mode)` 自体は純粋関数として出力を持たせない

## 採用する構成

### Rust

- `perft(board, depth, mode)`
  - `mode=1` をデフォルトとし、モード差を切り替えられる公開 API
- `PerftError`
  - まずは `InvalidMode` を持つ最小構成で実装する
- Perft 用の内部再帰 helper
  - モード差だけを切り替えられる形で再帰処理を閉じ込める
- 内部 mode enum
  - 公開 API の数値 mode を型安全に扱うための内部表現として実装する
- 長時間検証用 helper
  - ルート手ごとの進捗表示を行うデバッグ補助を用意する

### テスト

- 初期局面既知値テスト
  - モード1 / モード2 の深さ 0 から 8 の既知値を確認する
- 強制パス局面テスト
  - パス 1 回を含む局面でモード1 / モード2 の差を確認する
- 終局局面テスト
  - 終局局面でモードに関わらず 1 を返すことを確認する
- 長時間検証
  - 深さ 9 から 15 の既知値は `Makefile` 経由の手動コマンドで確認する

### 2. 強制パス局面の検証

- 合法手 0 件の側から始めても正しくパスが挿入されること
- モード1ではパスで深さを 1 消費し、モード2では消費しないこと
- パス後の相手番局面から通常どおり再帰すること

### 3. 終局局面の検証

- 終局局面では深さとモードに関わらずノード数 1 を返すこと

## 導入時の注意

- このステップでは石数集計や勝敗判定には進まない
- Perft は性能測定にも使えるが、このステップではまず正しさ確認を主目的とする
- 深さ 9 から 15 の確認は長時間検証として扱い、通常の `cargo test` には含めない
- 進捗表示は `make perft-long` などの手動確認コマンド側で扱い、通常テストには含めない

## このステップを先に行う理由

石数集計や勝敗判定は比較的局所的な処理だが、Perft はゲーム進行全体を一度に検証できる。
そのため、まず Perft で基礎ロジックの総合整合性を固めてから、後続ステップで結果集計系 API を追加するほうが不具合の切り分けがしやすい。
