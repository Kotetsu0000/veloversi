# Step 04: 着手適用と反転処理の実装

## このステップの目的

合法手生成の次の単位として、合法手を 1 手適用して次局面を作れるようにする。
この段階では強制パスや局面種別判定には進まず、通常着手に対する反転処理、石配置更新、手番反転を独立して完成させることを目的とする。

## このステップで行うこと

- `MoveError` 型を定義する
- 着手地点から反転対象ビットを求める内部処理を実装する
- `apply_move_unchecked` を実装する
- `apply_move` を実装する
- 上記 API を検証する Rust テストを追加する
- Perft を後続ステップで接続する前提をテスト計画に反映する

## 導入対象

- `MoveError`
- 着手地点から反転対象ビットを求める内部処理
- `apply_move_unchecked`
- `apply_move`

## 受け入れ条件

- [ ] `MoveError` が非合法手を表現できることを確認するテストが成功する
- [ ] 反転対象ビットを求める内部処理が 1 方向の反転を返すことを確認するテストが成功する
- [ ] 反転対象ビットを求める内部処理が複数方向の反転を返すことを確認するテストが成功する
- [ ] `apply_move_unchecked` が初期局面の合法手を正しく適用することを確認するテストが成功する
- [ ] `apply_move` が石配置更新、反転処理、手番反転を行うことを確認するテストが成功する
- [ ] `apply_move` が非合法手を拒否することを確認するテストが成功する
- [ ] `apply_move` 後も黒白ビットが重ならないことを確認するテストが成功する
- [ ] `apply_move` 後の石数が期待どおりに更新されることを確認するテストが成功する
- [ ] `cargo test` が成功する

## 実装開始時点の不足

Step 03 時点では、合法手を列挙する機能までは実装されているが、実際に 1 手進めて次局面を構築する機能は未実装である。
このため、Perft や局面種別判定に進む前に、通常着手の適用処理を独立して実装する必要がある。

## 実装方針

- 着手適用は `Board` と `Move` を入力に取り、新しい `Board` を返す
- `MoveError` は仕様書に合わせて `IllegalMove`, `PassNotAllowed`, `TerminalBoard` を持つ
- Step 04 では通常着手のみを扱うため、このステップで実際に返すのは `IllegalMove` のみとする
- `apply_move` は非合法手を検出した場合に `MoveError` を返す
- `apply_move_unchecked` は合法手である前提の内部高速関数として実装する
- 反転対象ビットを求める処理は公開 API にせず、private helper として実装する
- 反転対象ビットの計算は合法手生成と同じ盤面インデックス規約 `0 = A1`, `7 = H1`, `63 = H8` を前提に扱う
- 実装方針の具体化では `ref` 配下の参考実装を参照する
- まずは通常着手のみを対象にし、強制パスや終局判定は後続ステップへ分離する
- Perft は `合法手生成 + 着手適用 + パス処理` の総合検証として有効なので、このステップでは計画に接続するところまでとし、実際の実装と既知値照合は後続ステップで行う

## 採用する構成

### Rust

- `MoveError`
  - 仕様書に合わせて `IllegalMove`, `PassNotAllowed`, `TerminalBoard` を持つ
- 反転対象ビットを求める内部処理
  - 着手地点に対して裏返る相手石の集合を返す private helper として実装する
- `apply_move_unchecked`
  - 合法手前提で高速に次局面を作る
- `apply_move`
  - 公開 API として合法性を検査してから次局面を返す

### テスト

- 初期局面着手テスト
  - 初期局面の代表着手で石配置と手番更新を確認する
- 反転方向テスト
  - 1 方向および複数方向の反転を確認する
- 異常系テスト
  - 非合法手や既に石があるマスを拒否できることを確認する
- 整合性テスト
  - 着手後も黒白ビットが重ならず、石数が期待どおりであることを確認する
- Perft テスト計画
  - 着手適用の実装後に、パス処理を含む後続ステップで既知値照合へ接続できるようにする

## 検証項目

### 1. 初期局面の着手適用

- 初期局面の 1 手について着手地点に石が置かれること
- 挟まれた相手石が反転すること
- 手番が相手へ切り替わること

### 2. 方向別反転の検証

- 横、縦、斜めの各方向で反転対象を正しく求められること
- 1 手で複数方向を同時に反転できること

### 3. 異常系の検証

- 既に石があるマスへの着手を拒否すること
- 相手石を挟めない着手を拒否すること
- `apply_move_unchecked` は合法手前提でのみ使うこと

### 4. 整合性の検証

- `apply_move` 後も黒白ビットが重ならないこと
- 着手側の石数が `1 + 反転数` だけ増えること
- 相手側の石数が `反転数` だけ減ること

### 5. Perft を使った検証方針

- 着手適用が入ったことで、Perft を総合検証へ接続する前提が整うこと
- ただし Step 04 では強制パスをまだ扱わないため、Perft の実装と既知値照合は後続ステップで行う

### 6. 実装完了時の確認コマンド

- `make check` が成功すること
- `make coverage-check` が成功すること
- `make mutants` を実行し、結果を確認すること

## 導入時の注意

- このステップでは強制パスを実行しない
- このステップでは終局判定を行わない
- 着手後に相手番の合法手が 0 件かどうかは別ステップで扱う
- Python バインディングへの露出は Rust 側の API が安定した後続ステップで扱う

## このステップを先に行う理由

Perft、局面種別判定、探索は、着手適用が正しく動いていないと信頼できる検証ができない。
そのため、まず通常着手の反転処理と盤面更新を独立して完成させ、後続ステップでパス処理や総合検証の不具合と切り分けられる状態を作る。
