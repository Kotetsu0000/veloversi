# Step 03: 合法手生成の実装

## このステップの目的

盤面型の次の単位として、現在の手番で打てる合法手を列挙できるようにする。
この段階では着手適用やパス処理には進まず、局面に対して「どこに打てるか」を正しく判定する機能を独立して完成させることを目的とする。

## このステップで行うこと

- `Move` 型を定義する
- `LegalMoves` 型を定義する
- `generate_legal_moves` を実装する
- Perft による合法手生成の検証方針を取り込む
- 初期局面に対する合法手生成テストを追加する
- 代表局面に対する合法手生成テストを追加する
- 合法手が存在しない局面のテストを追加する

## 導入対象

- `Move`
- `LegalMoves`
- `generate_legal_moves`

## 受け入れ条件

- [x] `Move` が盤面インデックス `0..63` の着手位置を表現できることを確認するテストが成功する
- [x] `LegalMoves` が合法手のビット集合と件数を保持することを確認するテストが成功する
- [x] `generate_legal_moves` が初期局面の合法手を返すことを確認するテストが成功する
- [x] `generate_legal_moves` が代表局面の合法手を返すことを確認するテストが成功する
- [x] `generate_legal_moves` が合法手なし局面で空集合を返すことを確認するテストが成功する
- [x] `LegalMoves::count` が `bitmask` 中のビット数と一致することを確認するテストが成功する
- [x] Perft の初期局面既知値を後続ステップで使う検証方針がテスト計画に反映されている
- [x] `cargo test` が成功する

## 実装開始時点の不足

Step 02 時点では盤面型と基本整合性は実装されているが、局面から合法手を導く機能は未実装である。
このため、着手適用に進む前に、現在局面に対する合法手列挙を独立して実装する必要がある。

## 実装方針

- 合法手生成は `Board` を入力に取り、手番側の合法手だけを返す
- 戻り値は `LegalMoves { bitmask, count }` とする
- パスは合法手として含めない
- 盤面インデックス規約は `0 = A1`, `7 = H1`, `63 = H8` を共有する
- まずは正しさを優先し、最初から SIMD 最適化は行わない
- 実装方針の具体化では `ref` 配下の参考実装を参照する
- 着手適用や反転処理の本実装とは切り離して、合法手判定だけを先に完成させる
- Perft は `合法手生成 + 着手適用` の総合検証として有効なので、このステップでは既知値をテスト計画へ組み込むところまでとし、実際の Perft 実装と検証は後続ステップで行う

## 採用する構成

### Rust

- `Move`
  - 着手先 1 マスを表す値型として実装する
- `LegalMoves`
  - 合法手のビット集合と件数を保持する
- `generate_legal_moves`
  - 8 方向を調べて合法手を列挙する

### テスト

- 初期局面テスト
  - 標準初期局面の合法手 4 つを確認する
- 代表局面テスト
  - 複数方向の反転可能性がある局面を確認する
- 合法手なしテスト
  - 合法手が 0 件の局面を確認する
- Perft テスト計画
  - 初期局面の既知値を使って後続ステップで総合検証できるようにする
  - Step 03 では Perft 自体はまだ実装しない

## 検証項目

### 1. 初期局面の検証

- 黒番の合法手が 4 件であること
- 合法手の位置が仕様どおりであること
- `count` と `bitmask.count_ones()` が一致すること
- Perft の深さ 1 と 2 の既知値を参照し、後続ステップで総合検証に接続できること

### 2. 代表局面の検証

- 1 方向だけでなく複数方向をまたぐ合法手判定ができること
- 手番が変わると合法手集合も変わること

### 3. 合法手なし局面の検証

- `bitmask == 0` を返すこと
- `count == 0` を返すこと
- パスを行動として含めないこと

### 4. Perft を使った検証方針

- Egaroucid の技術解説にある初期局面の既知値を参照する
- まず深さ 1 = 4、深さ 2 = 12 を最低限の確認対象にする
- 着手適用実装後は Perft により合法手生成と着手処理をまとめて検証する

## 導入時の注意

- このステップでは石を裏返したり盤面を更新したりしない
- パス処理の実行自体は次以降のステップで扱う
- move ordering や探索最適化は対象外とする
- Perft はこのステップ単独では完結せず、着手適用が入る次以降のステップで実装と検証を行う

## このステップを先に行う理由

着手適用は「どの手が合法か」が正しく定義されていないと検証できない。
そのため、まず合法手生成だけを独立して完成させ、後続ステップで着手処理の不具合と切り分けられる状態を作る。
