# Step 02: 盤面型と基本整合性の実装

## このステップの目的

ゲームロジック実装の最初の単位として、盤面を安全に保持し検証できる最小構成を実装する。
この段階では合法手生成や着手適用には進まず、後続のロジックが依存する盤面表現と基本 API を確立することを目的とする。

## このステップで行うこと

- `Color` 型を定義する
- `Board` 型を定義する
- `BoardError` 型を定義する
- `Board::new_initial` を実装する
- `Board::from_bits` を実装する
- `Board::to_bits` を実装する
- `Board::occupied_bits` を実装する
- `Board::empty_bits` を実装する
- `Board::validate` を実装する
- 上記 API を検証する Rust テストを追加する

## 導入対象

- `Color`
- `Board`
- `BoardError`
- `Board::new_initial`
- `Board::from_bits`
- `Board::to_bits`
- `Board::occupied_bits`
- `Board::empty_bits`
- `Board::validate`

## 受け入れ条件

- [ ] `Color` の黒番・白番表現を確認するテストが成功する
- [ ] `Board` が `black_bits`, `white_bits`, `side_to_move` を保持することを確認するテストが成功する
- [ ] `Board::new_initial` が標準初期局面を返すことを確認するテストが成功する
- [ ] `Board::from_bits` が黒白ビットの重なりを拒否することを確認するテストが成功する
- [ ] `Board::to_bits` が内部状態を崩さず返すことを確認するテストが成功する
- [ ] `Board::occupied_bits` が `black_bits | white_bits` と一致することを確認するテストが成功する
- [ ] `Board::empty_bits` が空きマスのビット集合を返すことを確認するテストが成功する
- [ ] `Board::validate` が基本整合性のみを検査することを確認するテストが成功する
- [ ] 初期局面の検証テストが成功する
- [ ] 異常系の検証テストが成功する
- [ ] `cargo test` が成功する

## 実装開始時点の不足

Step 01 時点では、検証基盤は整っているが、盤面型とゲームロジックの基礎データ構造は未実装である。
このため、以後の合法手生成や着手処理に進む前に、盤面表現を固める必要がある。

## 実装方針

- 盤面は仕様どおり `black_bits: u64`, `white_bits: u64`, `side_to_move: Color` で保持する
- 盤面インデックス規約は `0 = A1`, `7 = H1`, `63 = H8` を前提に扱う
- `Board::validate` では到達可能性は扱わず、基本整合性のみを検査する
- `Board::from_bits` では最低限の不正入力だけを拒否する
- まず Rust 側の純粋ロジックとして実装し、Python 公開は後続ステップで扱う

## 採用する構成

### Rust

- `Color`
  - 手番を表す列挙型として実装する
- `Board`
  - ビットボード 2 枚と手番で盤面を表現する
- `BoardError`
  - 不正盤面入力や検証失敗を表現する

### テスト

- 初期局面テスト
  - 石数、手番、ビット配置を確認する
- ビット演算テスト
  - `occupied_bits` と `empty_bits` の整合を確認する
- 異常系テスト
  - 重複ビットや不正入力を拒否できることを確認する

## 検証項目

### 1. 初期局面の検証

- 黒石 2 個、白石 2 個であること
- 初期手番が黒であること
- `validate()` が成功すること

### 2. ビット演算の検証

- `occupied_bits()` が `black_bits | white_bits` と一致すること
- `empty_bits()` が占有ビットの補集合と一致すること
- `occupied_bits() & empty_bits() == 0` であること

### 3. 異常系の検証

- 黒白ビットが重複する場合に `from_bits()` が失敗すること
- `validate()` が不正な盤面を検出できること
- `to_bits()` が入力状態を保持すること

## 導入時の注意

- このステップでは合法手生成や着手適用まで含めない
- 盤面の到達可能性判定は扱わない
- Python バインディングへの露出は次以降のステップで行う

## このステップを先に行う理由

合法手生成や着手処理は、盤面表現が安定していないと正しく検証できない。
そのため、まず盤面型と基本整合性の API を独立して完成させ、後続ステップでロジック不具合の切り分けができる状態を作る。
