# Step 17: `ref` AI 移植 Phase 1

## このステップの目的

このステップでは、`ref` にある AI 実装を段階的に Rust 側へ寄せていく最初の段階として、探索の最小核を移植する。

方針は次のとおり。

- `ref` にあるものを段階的に寄せる
- `ref` に関係ない独自探索は作らない
- 依存の少ない中核から移す
- Python API は後段に回し、まず Rust API を安定させる

## version 方針

- `ref` AI が完成するまでは `0.0.1` を維持する
- `ref` AI が完成した段階で `0.1.0` を出す

## このステップで行うこと

- `ref` AI の入口となる探索実装を確認する
- 依存関係を最小単位で分解する
- 探索の最小核を Rust に移植する
- `best_move` と `score` を返す Rust API を追加する
- `ref` 由来の最小評価導線をつなぐ
- `make check` が通る状態まで整える

## このステップの対象範囲

### 対象

- `ref` AI の探索の最小核
- 深さ固定探索
- `best_move`
- `score`
- Rust API

### 対象外

- 独自評価器
- 独自探索アルゴリズム
- Python API の本格公開
- build feature の有効化実装
- exact solver
- PV の充実
- 時間管理
- `ref` AI 入り install 導線

## 固定した前提

- 評価器は `ref` にある最小依存だけを使う
- 独自の簡易評価器は作らない
- Python API は同 step 内でも後段。まず Rust API を先に作る
- build feature は設計だけ意識し、この step では実装しない
- `ref` の探索入口と評価入口を起点に依存を辿る

## 受け入れ条件

- [ ] `ref` AI 移植の入口ファイルが特定されている
- [ ] 探索の最小核が Rust に入っている
- [ ] `best_move` を返す Rust API がある
- [ ] `score` を返す Rust API がある
- [ ] `ref` に無い独自探索を追加していない
- [ ] `make check` が成功する

## 実装方針

- まず `ref` の探索入口と評価入口だけを読む
- 枝葉の utility を先に広く移さない
- 1 回で全移植せず、最小で動く単位を優先する
- `ref` の構造と責務を崩さない
- 比較しやすいよう、API も役割単位で最小に留める

## このステップを先に行う理由

基盤機能、feature、random_play、release 導線まで揃ったため、次はこのライブラリのもう一つの主目的である `ref` AI 再現に入る段階にある。
ここで最小核から着手することで、以後の段階的移植をぶらさず進めやすくする。
