# Step 07: Perft 高速化の実装

## このステップの目的

Step 06 で正しさを確認した Perft を、深さ 15 の長時間検証が現実的な時間で回るように高速化する。
この段階では新しいゲーム機能は追加せず、Perft 実行経路の無駄を減らして測定可能な改善を得ることを目的とする。

## このステップで行うこと

- Perft 実行経路のホットパスを見直す
- `board_status` 依存を減らす
- 必要なら Perft 専用の軽量 helper を追加する
- 必要なら末端深さの特殊化を追加する
- `make perft-long` 向けのルート手単位並列化を検討・実装する
- 高速化後の Perft 実行時間を確認する

## 導入対象

- Perft 実行経路の最適化
- 必要に応じた Perft 専用 helper
- 必要に応じた `make perft-long` の改善
- 必要に応じたルート手単位並列化

## 受け入れ条件

- [ ] Step 06 の Perft 既知値テストが引き続き成功する
- [ ] `make check` が成功する
- [ ] `make coverage-check` が成功する
- [ ] `make mutants` を実行し、結果を確認する
- [ ] `make perft-long` が実行できる
- [ ] 高速化前後で `make perft-long` の実行時間を比較できる
- [ ] 少なくとも 1 つ以上のホットパス最適化について、改善内容を README か計測メモで説明できる

## 実装開始時点の不足

Step 06 時点では Perft の正しさは確認できるが、深さ 11 以降の実行時間が長く、深さ 15 の既知値確認を日常的に回すには重い。
このため、Perft 専用の実行経路を見直して、不要な判定や重複計算を減らす必要がある。

## 実装方針

- まずは `make perft-long` の実行時間を基準値として把握する
- Perft では直前に得た合法手情報を活用し、不要な `board_status` 呼び出しを避ける
- 必要なら `perft` 専用に局面遷移を軽くする helper を用意する
- 深さ 1 などの末端ケースは特殊化して分岐コストを減らす
- `perft(board, depth, mode)` の公開 API と既知値の意味は変えない
- `perft(board, depth, mode)` 本体は単一スレッドの純粋関数として保つ
- 並列化は `make perft-long` 用の長時間検証 helper でのみ行い、ルート手単位に限定する
- ルート手単位並列化には `rayon` を採用する
- 実装方針の具体化では `ref` 配下の参考実装を参照する
- 高速化は必ず既知値一致で検証してから採用する

## 採用する構成

### Rust

- Perft ホットパス最適化
  - 既存 API を保ったまま内部処理だけを軽くする
- Perft 専用 helper
  - 必要な場合だけ追加し、通常 API から利用する
- 長時間検証用並列 helper
  - `rayon` を使ってルート手単位で部分木を並列に数える

### テスト / 計測

- 既知値回帰テスト
  - Step 06 の Perft 既知値が壊れていないことを確認する
- 実行時間確認
  - `make perft-long` の実時間を高速化前後で比較する
- 進捗表示確認
  - 長時間コマンドで進捗が見えることを確認する

## 検証項目

### 1. 正しさの回帰確認

- 初期局面の深さ 0 から 8 の既知値が引き続き一致すること
- 強制パス局面で mode 差が引き続き一致すること

### 2. 高速化確認

- 高速化前後で `make perft-long` の実行時間を比較できること
- 少なくとも体感ではなく数値で改善を説明できること

### 3. 運用確認

- `make perft-long` の進捗表示が長時間実行時の状態把握に使えること
- 並列化しても進捗表示と最終集計が破綻しないこと

## 導入時の注意

- このステップでは Perft の意味や既知値を変えない
- 速度のために API 仕様や mode の意味を変えない
- 読みやすさを壊しすぎる最適化は避ける
- 必要なら「まず簡単な最適化だけを入れる」方針で段階的に進める

## このステップを先に行う理由

Perft は以後のロジック変更の回帰検出にも使うため、正しいだけでなく、現実的な時間で実行できることに価値がある。
そのため、次の集計系 API や探索系実装へ進む前に、Perft の実用速度を上げておくと後続の検証効率が上がる。
