# Step 05: 強制パスと局面種別判定の実装

## このステップの目的

通常着手の実装に続いて、合法手が存在しない局面での強制パスと、局面が継続中・強制パス・終局のどれに当たるかを判定できるようにする。
この段階では Perft 本体や石数集計には進まず、ゲーム進行上の局面遷移判定を独立して完成させることを目的とする。

## このステップで行うこと

- `BoardStatus` 型を実装する
- `apply_forced_pass` を実装する
- `board_status` を実装する
- 上記 API を検証する Rust テストを追加する
- Perft を後続ステップで接続する前提をテスト計画に反映する

## 導入対象

- `BoardStatus`
- `apply_forced_pass`
- `board_status`

## 受け入れ条件

- [x] `BoardStatus` が `Ongoing`, `ForcedPass`, `Terminal` を表現できることを確認するテストが成功する
- [x] `apply_forced_pass` が合法手 0 件の局面でのみ手番を反転することを確認するテストが成功する
- [x] `apply_forced_pass` が石配置を変更しないことを確認するテストが成功する
- [x] `apply_forced_pass` が合法手あり局面で `MoveError::PassNotAllowed` を返すことを確認するテストが成功する
- [x] `apply_forced_pass` が終局局面で `MoveError::TerminalBoard` を返すことを確認するテストが成功する
- [x] `board_status` が合法手あり局面を `Ongoing` と判定することを確認するテストが成功する
- [x] `board_status` が現手番のみ合法手なし局面を `ForcedPass` と判定することを確認するテストが成功する
- [x] `board_status` が両者合法手なし局面を `Terminal` と判定することを確認するテストが成功する
- [x] `cargo test` が成功する

## 実装開始時点の不足

Step 04 時点では、通常着手を適用する機能までは実装されているが、合法手が 0 件の局面でどう遷移するか、および現在局面が継続局面か終局局面かを判定する機能は未実装である。
このため、Perft や探索に進む前に、パス処理と局面種別判定を独立して実装する必要がある。

## 実装方針

- `BoardStatus` は仕様書どおり `Ongoing`, `ForcedPass`, `Terminal` を持つ
- `apply_forced_pass` は石配置を変更せず、手番だけを反転する
- `apply_forced_pass` は合法手が存在する局面では `MoveError::PassNotAllowed` を返す
- `apply_forced_pass` は両者とも合法手がない終局局面では `MoveError::TerminalBoard` を返す
- `board_status` は `generate_legal_moves` を使って現手番と相手番の合法手有無を判定する
- 相手番の合法手確認では一時的に手番を反転した盤面を使う
- 強制パス局面と終局局面の固定テストケースは、実装時に素朴オラクルで条件を満たす局面を探索して確定させる
- 実装方針の具体化では `ref` 配下の参考実装を参照する
- Perft は `合法手生成 + 通常着手 + 強制パス` の総合検証として有効なので、このステップで前提を揃え、実際の実装と既知値照合は後続ステップで行う

## 採用する構成

### Rust

- `BoardStatus`
  - 現局面の進行状態を表す列挙型として実装する
- `apply_forced_pass`
  - 強制パス局面でのみ手番を反転した `Board` を返す
- `board_status`
  - 継続、強制パス、終局を判定して返す

### テスト

- 継続局面テスト
  - 初期局面など合法手がある局面を `Ongoing` と判定することを確認する
- 強制パス局面テスト
  - 素朴オラクルで確認した固定局面を使い、現手番に合法手がなく相手番に合法手があることを確認する
- 終局局面テスト
  - 素朴オラクルで確認した固定局面を使い、両者とも合法手がないことを確認する
- パス適用テスト
  - 強制パスで石配置が変わらず手番だけ反転することを確認する
- 異常系テスト
  - 合法手あり局面と終局局面で `apply_forced_pass` が適切なエラーを返すことを確認する
- Perft テスト計画
  - 強制パスまで揃った段階で、後続ステップの Perft 実装へ接続できるようにする

## 検証項目

### 1. 継続局面の検証

- 初期局面など合法手が存在する局面で `board_status()` が `Ongoing` を返すこと
- `apply_forced_pass()` がその局面では失敗すること

### 2. 強制パス局面の検証

- 現手番の合法手が 0 件であること
- 相手番の合法手が 1 件以上あること
- `board_status()` が `ForcedPass` を返すこと
- `apply_forced_pass()` が手番だけ反転した `Board` を返すこと

### 3. 終局局面の検証

- 現手番と相手番の両方で合法手が 0 件であること
- `board_status()` が `Terminal` を返すこと
- `apply_forced_pass()` が `MoveError::TerminalBoard` を返すこと

### 4. Perft を使った検証方針

- 強制パスまで実装されることで、Perft を成立させるための局面遷移が揃うこと
- Step 05 では Perft 自体はまだ実装せず、後続ステップで既知値照合を行うこと

### 5. 実装完了時の確認コマンド

- `make check` が成功すること
- `make coverage-check` が成功すること
- `make mutants` を実行し、結果を確認すること

## 導入時の注意

- このステップでは Perft 本体は実装しない
- このステップでは石数集計や勝敗判定には進まない
- `apply_forced_pass` は「合法手なしだが相手は打てる」局面だけを対象にする
- Python バインディングへの露出は Rust 側の API が安定した後続ステップで扱う

## このステップを先に行う理由

Perft や探索は、通常着手だけでなく強制パスも正しく扱えないと成立しない。
そのため、まず局面遷移として必要なパス処理と局面種別判定を独立して完成させ、後続ステップで総合検証や石数評価の不具合と切り分けられる状態を作る。
